# Roadmap

The build order for SecurMCP, sequenced across four phases so there's always something demoable at the end of each phase rather than a long stretch with nothing to show. Kept as a living checklist — update this file as items complete rather than letting it drift from reality.

See [`ARCHITECTURE.md`](./ARCHITECTURE.md) for what each item actually means and [`docs/adr/`](./docs/adr/) for the reasoning behind deferred items (Kubernetes, Terraform-first, OPA, etc.).

---

## Build Order — Four Phases

**Phase 1 — Core gateway (get something real working end-to-end)**
1. ~~Repo scaffold, Docker Compose skeleton, Postgres schema + Alembic migration for `audit_log` and `policies`~~ — **done**: the "policies" table is `policy_versions` per `ARCHITECTURE.md` §4.8 (YAML files remain the policy content store); `audit_log.signature` was nullable until item 11 made it NOT NULL (migration 0003).
2. ~~Session Manager + JSON-RPC Interceptor wired to a single hardcoded upstream server, full passthrough, no policy yet — includes the `lifespan`/SIGTERM subprocess-cleanup handler (with per-subprocess `ProcessLookupError`/`OSError` handling) and the Redis-TTL session idle timeout from day one, since both are cheap to add while there's only one subprocess type to manage and easy to forget once more moving pieces exist.~~ — **done**: client-facing transport is Streamable HTTP rather than SSE (the MCP spec deprecated standalone SSE in 2025-03; README tech-stack table updated); upstream is a per-session stdio subprocess set via `UPSTREAM_COMMAND`.
3. ~~Policy Engine (RBAC only for now) + Schema Pruner — identity-scoped `tools/list`~~ — **done**: RBAC is also enforced on every `tools/call` per §4.1's point-of-action principle (pruning is not the boundary); identity comes from a temporary unverified `X-SecurMCP-Identity` header until item 4 replaces it with key auth.
4. ~~Auth layer (API key → identity, hash-and-lookup, no HMAC/JWT)~~ — **done**: `X-SecurMCP-Key` verified on every request per §4.8 (unauthenticated → HTTP 401 before a subprocess spawns; a valid key can't ride another identity's session); plain unsalted SHA-256 per §4.8, which supersedes §6's "salted hashes" wording — salting is incompatible with hash-and-lookup and unnecessary for 256-bit random keys. Keys minted via `scripts/generate_api_key.py`. The Redis auth-failure counter is Phase 3, item 18 as scheduled.
5. ~~Audit log writer with hash chaining (Redis-cached `latest_audit_hash`, updated atomically via Lua script/`WATCH`-`MULTI`, to avoid a `SELECT MAX(seq)` on every write; signing comes in Phase 2) + basic verifier~~ — **done**: ALLOW is written *before* forwarding per §4.8's write-path paragraph (so `latency_ms`/result stay NULL in v1); `canonicaljson` pinned now rather than at item 9, since changing serializers mid-chain would invalidate earlier rows; a basic concurrency test landed here (item 23 keeps the expanded adversarial version).
6. ~~Parameter Validator on `tools/call`~~ — **done**: validates against a per-session schema cache filled from `tools/list` responses (item 7's shared TTL/ETag cache replaces it); a call for a never-listed tool fails closed as `DENY_VALIDATION`; sanitization strips traversal/null-byte/control-char patterns and records touched fields in the ALLOW audit payload. Hypothesis fuzzing lands with the expanded adversarial suite (item 23).
7. ~~**Cache invalidation (schema TTL, ETags, policy hot-reload)** — moved up from Phase 2: real MCP clients (Claude Desktop, Cursor) poll or re-fetch `tools/list` more than a naive baseline assumes, and caching bugs discovered late are hard to distinguish from "the core proxy is broken." Getting this right before Phase 2 hardening begins means Phase 2 testing debugs one thing at a time, not two at once.~~ — **done**: hot-reload via SIGHUP (`docker kill -s HUP`; broken reload keeps last-known-good; success writes `POLICY_ACTIVATED`); shared Redis schema cache per `server_id` (TTL 10 min, invalidated on `initialize`) with transparent gateway-issued `tools/list` re-fetch on miss — supersedes item 6's deny-on-never-listed, which now fires only if the re-fetch fails; §8's ETag is delivered as `_meta.etag` on the `tools/list` result (per-message conditional HTTP semantics don't exist over the streamed transport). The re-fetch hook is where item 9's drift check slots in.
8. ~~`sample_target/overscoped_server.py` + first recorded demo (schema pruning only)~~ — **done**: overscoped server is the compose default upstream (stdio, spawned in the gateway container — no separate service needed until `rogue_server`'s admin endpoint in Phase 2); `scripts/run_demo.py` is the verified, record-ready driver (mints keys into gitignored `policies/demo-policy.yaml`, shows developer-pruned vs admin-full lists + audit receipts); the actual screen recording is an operator step. **Phase 1 complete.**

**Phase 2 — Drift detection, hardening, and proof it's fast enough**
9. ~~Drift Detector with severity classification (Low/Medium/High/Critical) and `canonicaljson` (pinned version) for RFC 8785 canonicalization, plus the key-reordering smoke test — this is where "block on any change" becomes "block on the changes that matter."~~ — **done**: baselines in a new `tool_baselines` table (migration 0002; first sighting = trust anchor, observed drift deduped per hash); High/Critical blocks at `tools/call` as pipeline stage 5 (`DENY_DRIFT`); re-approval via `POST /admin/tools/{server_id}/{tool_name}/approve`, authenticated by a new `admin: true` identity flag in the policy YAML (v1's admin authn model, reused by Phase 3 admin endpoints); rename heuristic compares shape hashes with cosmetic `title` fields stripped (SDK servers embed the function name in titles); unclassifiable schema changes fail closed as High.
10. ~~Replay Guard (nonce + timestamp + Redis dedup).~~ — **done**: nonce/timestamp travel in `tools/call` `params._meta` under `securmcp/nonce` (client-generated UUID) and `securmcp/timestamp` (Unix epoch seconds) — MCP reserves `_meta` for out-of-band metadata, and the SDK's `call_tool(meta=...)` sends it natively; runs as the first check in `_authorize_call` (§4.2 stage 1, before RBAC); nonce dedup via Redis `SET NX` with TTL 2× `REPLAY_WINDOW_SECONDS` (default 30, i.e. the ±30s half-width — 2× strictly outlives a nonce's validity); missing/malformed fields, an out-of-window timestamp, and Redis being unreachable all fail closed as `DENY_REPLAY`; the pair is stripped from `_meta` before an allowed call is forwarded upstream (other `_meta` content passes through).
11. ~~ECDSA signing added to the audit log; verifier daemon checks signatures and chain math incrementally from a `last_verified_seq` checkpoint, not a full scan from `seq=1`.~~ — **done**: P-256 keypair minted once via `scripts/generate_signing_key.py` into gitignored `secrets/` (mounted read-only; Secrets Manager injection is the Phase 4 deployment story) — a gateway with no usable private key fails startup (§5); every row's `curr_hash` gets a DER ECDSA-SHA256 signature; migration 0003 makes `audit_log.signature` NOT NULL (completing item 1's deferral — it fails by design on a chain with unsigned rows, which cannot be signed retroactively; wipe dev/demo chains first); the checkpoint is a single-row Postgres table (`audit_verifier_checkpoint`, not Redis — per ADR-003 losing it would force the O(n) rescan this item exists to avoid); `scripts/audit_verifier_daemon.py` runs as a compose sidecar (public key only) or under cron via `--once`, never advances the checkpoint past a confirmed break, and never blocks live traffic (§5 detective control); alerting is `logger.error` until items 13/25; `scripts/verify_audit_chain.py` (full scan) now also checks signatures when the public key is present.
12. ~~Performance benchmark suite + first published latency numbers in the README, including the `tools/list` payload-size reduction metric.~~ — **done**: `tests/benchmarks/run.py` (`python -m tests.benchmarks.run`), a small asyncio harness rather than locust — §9 allowed either and this adds no dependency; one in-process gateway (the integration suite's `running_gateway`) serves every scenario including all concurrency levels; cold-cache = Redis DEL of the schema key before each timed call, forcing the §8 transparent re-fetch + drift check per call; direct baseline is a stdio MCP client straight at `overscoped_server.py`; the module isn't pytest-collectable, so the default suite stays fast with zero config; per-run artifacts land in gitignored `tests/benchmarks/reports/`, and the committed numbers live in the README's new Performance section stamped with the commit + date they were measured against (§9's staleness rule); CI benchmark-on-merge is item 15 as scheduled.
13. ~~Structured logging (structlog) end to end.~~ — **done**: one `configure()` in `services/gateway/logging_config.py` using the `structlog.stdlib.ProcessorFormatter` bridge, so foreign stdlib records (uvicorn, sqlalchemy) render as the same JSON shape as structlog's own — JSON always, no dev console renderer; §7's decision line now fires for ALLOW too, not just denies, with correlation id = session id via a logger bound at `Interceptor` construction (interceptor calls run in the per-session pump task, out of reach of request-scoped contextvars — the Interceptor gained a `session_id` field for this); the verifier daemon's `logging.basicConfig` replaced by the same `configure()`, covering both compose containers; `audit_verifier` alerting stays `logger.error` (now structured) until item 25's Prometheus counters.
14. ~~`sample_target/rogue_server.py` with a real `POST /_admin/apply_mutation` endpoint (no timer) + updated demo recording (schema pruning + drift blocking, admin mutation visible on-screen).~~ — **done**: rogue serves `send_email`/`read_inbox`/`delete_mailbox` (the extra two give the pruning beat a single-upstream home) via the low-level `Server` API reading a state file on every `tools/list`, so a live session sees the mutation without reconnecting; the admin endpoint runs as its own compose service (`rogue`, port 9800, stdlib `http.server`) sharing the gitignored `.rogue-state/` bind mount with the gateway's per-session subprocesses; the mutated `bcc` is **required** — an optional param classifies Medium and wouldn't block; `scripts/run_demo.py` drives the full narrative and now resets the dev audit chain + drift baselines at start (a stale baseline makes the *benign* schema read as drift); `tests/integration/test_rogue_demo.py` verifies the whole arc through the real endpoint before anyone records it; the screen recording itself remains an operator step.
15. ~~CI/CD pipeline (lint, typecheck, test, coverage gate, benchmark-on-merge, build).~~ — **done**: `.github/workflows/ci.yml` on push + PR runs lint (`ruff check` + `ruff format --check` — landing the latter took a one-time mechanical `ruff format .` pass over 14 files), `mypy services/`, `pytest --cov=services --cov-fail-under=80` (pytest-cov added to dev deps; coverage sits at ~93%, no omit config needed) against `postgres:16`/`redis:7` service containers so the integration/adversarial suites actually run rather than skip — no secrets in CI, the fixtures mint ephemeral signing keypairs — and a verify-only `docker build .`; the benchmark job is gated to pushes on `main` per §9, runs `python -m tests.benchmarks.run 100` (N=100 vs the local default 1000 to keep a shared runner practical; the fixed 10/50/100-session concurrency scenarios are unaffected) and uploads `tests/benchmarks/reports/` as a 30-day artifact for regression tracking — runner numbers never rewrite the README's measured Performance section; `release.yml` on `v*` tags builds and pushes `ghcr.io/<owner>/securmcp` tagged with the semver + `latest` via `GITHUB_TOKEN` (`packages: write`) — GHCR only, ECR stays a post-MVP option per §12. **Phase 2 complete.**

**Phase 3 — Risk-aware, expressive policy, and the standout features**
16. ~~Risk Engine v1 (factor-list scoring: tool sensitivity, blast radius, business hours, call frequency, drift-in-review) plus the risk decay feedback loop (per-identity/tool calibration counter on approval, behavioral factors only, never the static sensitivity tier).~~ — **done**: five-factor weighted sum in `services/gateway/risk_engine.py` (a fixed `evaluate(ctx)` function list per §4.8 — no plugin loader, no ML) wired as pipeline stage 6 between drift and param validation; static inputs come from a new optional `risk:` policy block (`tool_sensitivity` tiers low/medium/high/critical → 10/20/30/45; `protected_repos` fnmatch patterns matched against every string argument value), business hours are Mon–Fri 09:00–18:00 **UTC** via settings (no per-identity timezone in v1), and the behavioral factors are call frequency (Redis `risk:freq:*` rolling counter) and drift-in-review (unresolved `observed_*` on `tool_baselines` — Low/Medium drift elevates risk without blocking). Thresholds per §4.8: <40 continue, 40–69 `CHALLENGE` (terminal JSON-RPC error; real step-up auth stays Phase 4), 70–90 `HUMAN_APPROVAL_REQUIRED`, >90 `DENY_RISK`; a scoring exception is score 100 (§5 fail-closed); score + factors travel in the canonical Decision and land in the audit row. The approval lifecycle lives in a new `approvals` table (migration 0004; restart-durable, with a startup sweep expiring stale pending rows): the held call is *not* auto-retried — after `POST /admin/approvals/{approval_id}/approve` (admin-flag auth, same model as drift re-approval) the client re-invokes with the id in `params._meta["securmcp/approval_id"]`; redemption is one-time (reuse → `DENY_REPLAY`), TTL 15 min default (lapse → `EXPIRED`), and the arguments hash stored at request time is recomputed at redemption — a mismatch is the TOCTOU case, `DENY_APPROVAL_MISMATCH`. Risk decay fires on the approve endpoint: `INCRBY` on the per-(identity, tool) `risk:decay:*` offset (default 5), discounting only the behavioral subtotal (floored at 0), never the sensitivity tier.
17. ~~ABAC conditions layered onto the policy engine (embedded expression evaluator; explicitly no loops/recursion/arbitrary code; missing-attribute references evaluate the whole condition as not-satisfied, never injected as a raw boolean into a `not`).~~ — **done**: `conditions: list[str]` on each ServerGrant, compiled at policy load in `services/gateway/abac.py` — `ast.parse(mode="eval")` plus a strict node whitelist (and/or/not, the six comparison operators, parentheses, two-segment dotted paths rooted at identity/tool/context/risk, string/number/bool literals; anything else fails validation, so a bad condition fails startup and a SIGHUP reload keeps last-known-good) — evaluation is a hand-written walk per ADR-004, no eval/exec anywhere. Attribute sources: `identity.*` from a new optional `attributes:` map on the identity (identity.id from the id itself), `tool.name`/`tool.server_id` from the call, `context.hour` derived UTC from the stage-1-validated replay timestamp, `risk.score` from stage 6. The §4.2-vs-§4.8 ordering conflict (ABAC is stage 4, risk.score doesn't exist until stage 6) is resolved by splitting each grant's conditions at load: conditions with no `risk.*` reference run at stage 4 (after RBAC, before drift); `risk.*` conditions run immediately after scoring but **before** the 40/70/90 threshold mapping, so a failing `risk.score < 60` is `DENY_ABAC` even when the score would otherwise CHALLENGE or hold for approval (stage 4 precedence preserved). On an approval-redemption retry there is no fresh score, so `risk.*` conditions are deliberately skipped — a human approved that exact arguments-hash — while non-risk conditions still apply. Missing-attribute rule per §4.8: all referenced paths are resolved before any combinator logic, so one unresolvable path makes the whole condition not-satisfied (a `not(...)` can never invert it) and writes a `POLICY_ERROR` audit row; type-mismatch comparisons and evaluator exceptions take the same POLICY_ERROR + `DENY_ABAC` fail-closed path. ALL conditions on the RBAC-matched grant must hold (AND); pruning stays RBAC-only per §4.1.
18. ~~Richer risk telemetry: prior-denial-rate, drift-history, and an auth-failure counter added to the Auth Layer.~~ — **done**: three new factor functions appended to the fixed `FACTORS` list (same `evaluate(ctx)` interface, no engine changes). **Prior-denial-rate** is a count threshold, not a ratio: a per-identity Redis rolling counter (`risk:denials:*`, default window 600s) bumped best-effort by the interceptor on every `DENY_*` terminal — CHALLENGE/HUMAN_APPROVAL_REQUIRED don't count — firing +25 above the threshold (default >3). **Auth-failures** is the §4.8-literal *single gateway-wide* counter (`risk:auth_failures`, window 300s), since a failed key lookup by definition resolves to no identity: `auth.resolve_identity_tracked()` wraps the pure lookup and bumps it on a *wrong* key (a missing header isn't a stuffing signal) at all three call sites, and the factor (+20 above >5) fires for every identity while the spike is live — per-IP attribution is deliberately out of scope for v1. **Drift-history** (+15 at ≥2 events in 7 days, "changed shape twice in the last week") reads the audit log via `DriftDetector.recent_drift_count()` — counting `DRIFT_*` rows per (server, tool) in the window (existing `idx_audit_event`, no migration, no new state), which is what makes the "even if already re-approved" property hold: re-approval clears `tool_baselines.observed_*` but audit rows are forever. Failure posture: telemetry *writes* fail soft (a Redis hiccup while counting a deny or a 401 never disturbs the deny/401 itself); scoring *reads* stay inside `RiskEngine.score()`, which raises → score 100 (§5 fail-closed). Decay boundary per §4.8's explicit list: `BEHAVIORAL_FACTORS` gains only `prior_denial_rate`; `auth_failures` and `drift_history` are not decayable — approving one call shouldn't erase a gateway-wide stuffing signal or a tool's instability record. The §11 canary (high tier + protected + off-hours = 85) is untouched.
19. ~~Policy versioning (version stamping, revision snapshots, rollback) plus the `--diff-policy` terminal mode and `--html` side-by-side diff flag (`difflib.HtmlDiff`, no new dependency).~~ — **done**: every activation (startup, SIGHUP, rollback) is recorded by `services/gateway/policy_versions.py` — an append-only snapshot of the exact file bytes at `policies/revisions/v{n}.yaml` (never re-serialized; gitignored, `.gitkeep` stays) plus a `policy_versions` row (`activated_by` = `startup` / `operator` / the admin's identity id), finally populating item 1's table. Forward activation is monotonic: the YAML `version` must be strictly greater than the highest ever recorded; the same (version, content_hash) pair is an idempotent no-op; the same version with different content is rejected fail-closed (startup fails, SIGHUP keeps last-known-good via the new `load_candidate()`/`swap()` split — record before swap). The POLICY_ACTIVATED audit row is written on *every* boot, not just first-seen (deviation from the item's original sketch): a restart that reverts an unsynced rollback must be visible in the chain. Rollback (`POST /admin/policy/rollback/{version}`, admin-flag auth) re-activates a prior snapshot after verifying its bytes still match the recorded `content_hash` (mismatch = tampered snapshot, 409) — **in-memory only**: `./policies` stays mounted `:ro` (a compromised gateway can't persist a policy backdoor; only the new rw `./policies/revisions` submount is writable for snapshots), so POLICY_FILE on disk keeps the newer version until the operator updates it, and the endpoint logs a warning saying exactly that. `scripts/verify_audit_chain.py --diff-policy v3 v4` prints a unified diff of two snapshots (no DB needed); `--html` writes a standalone side-by-side `policy-diff-v3-v4.html` via stdlib `difflib.HtmlDiff` — chain verification stays the no-flag default. Dev resets (`run_demo.py`, benchmarks, the integration `clean_audit` fixture) now also truncate `policy_versions`; the demo reset additionally clears stale revision snapshots (it re-mints keys into the same version number every run).
20. ~~**Decision Explanation** (`GET /admin/decisions/{id}`, `POST /admin/decisions/explain`) — build this right after the Risk Engine and Policy Engine exist, since it's mostly just exposing data they already compute.~~ — **done**: both entry points live in `services/gateway/decision_explainer.py` behind the standard admin-flag auth. `GET /admin/decisions/{seq}` ({seq} = the `audit_id` clients already receive) reconstructs the canonical §4.3 Decision from the audit row; terminal decision writes now persist `matched_rules` in the payload so new rows are faithful, with a per-event-type fallback for older rows; non-decision rows (SESSION_START, TOOLS_LIST, DRIFT_*, POLICY_ACTIVATED, POLICY_ERROR) are 404, not a bent Decision. `POST /admin/decisions/explain` (`{identity, tool, arguments, context}`) dry-runs the §4.2 pipeline against the current in-memory policy — same stage order and fail-closed logic minus the replay guard (a hypothetical call has no nonce) and minus every side effect: no audit rows (including ABAC POLICY_ERROR), no approvals row, no upstream traffic, and risk scoring runs `dry_run=True` (frequency counter read+1, never INCRed). `context.hour` precedence: explicit `hour` → UTC hour of `timestamp` → now(UTC). Param validation reads only the shared schema cache — a cold cache fails closed as DENY_VALIDATION with a reason saying a live call may still succeed via the §8 re-fetch. The Decision gained the §4.8 `alternative` field: the 40/70/90 threshold-mapped outcome of `risk_score`, set only when it differs from the actual outcome (e.g. an ABAC `risk.score` deny at 74 → `human_approval_required`), null otherwise.
21. ~~**Policy Simulation Mode** — build alongside Decision Explanation; both reuse the same underlying decision/audit data.~~ — **done**: `POST /admin/policy/simulate` (admin-flag auth, `services/gateway/policy_simulator.py`) with the two §4.8 modes, mutually exclusive (400 otherwise): `{"candidate_version": N, "replay_window": "YYYY-MM-DD..YYYY-MM-DD"}` replays the window's decisions against revision N and reports `would_now_deny`/`would_now_require_approval`/`newly_allowed`/`unchanged`; `{"compare_versions": [old, new], ...}` replays through both and reports `new_denials`/`new_approvals`/`changed_risk_scores`/`changed_explanations` — plus `total_replayed` and up to 10 `sample_diffs` either way. The replay set is every decision row (ALLOW, the DENY_* family, CHALLENGE, HUMAN_APPROVAL_REQUIRED, EXPIRED — not APPROVED, which would double-count the row it redeemed) in the inclusive UTC date window with a tool name and payload `arguments`, oldest first, capped at 10k rows. Candidate engines load from the item-19 revision snapshots (`snapshot_path` + `load_bytes`), verified against the recorded `content_hash` (tampered snapshot = 409, unrecorded/missing = 404) and never activated — the endpoint is fully read-only: each row re-runs `decision_explainer.explain_call` (dry-run risk scoring, no audit rows) with `context.timestamp` = the row's epoch so business-hours ABAC evaluates the historical hour, and with the new `validate_params=False` — schemas aren't part of the policy, so stage 7 is skipped rather than letting schema-cache state masquerade as policy changes. To make denied calls replayable, the interceptor now persists raw arguments on deny-family payloads too (previously ALLOW-only); rows from before this item lack them and are skipped, so `newly_allowed` only becomes meaningful for traffic recorded from here on. v1 limitations: drift baselines and Redis telemetry are *current* state, not time-traveled, and the 10k cap bounds the endpoint. §11's hand-computed `would_now_deny` test runs end to end in `tests/integration/test_policy_simulation.py` (plus a compare-mode hand count, side-effect-free assertion, and the 400/403/404/409 paths).
22. ~~Component, deployment, and data-flow diagrams; multi-server trust domain discussion written up in `ARCHITECTURE.md` (documented, not built).~~ — **done** (docs only, no code diff): §4.1 sequence diagram rewritten to the actual §4.2 stage order (the old sketch ran param validation *before* risk and had no drift stage on `tools/call`), now showing the CHALLENGE / HUMAN_APPROVAL_REQUIRED + approval-redemption paths, the `tools/list` cache→drift→prune path (explicitly not the full pipeline), and Streamable HTTP as the client transport; §4.4 component diagram redrawn 1:1 against `services/gateway/` module names with pipeline stages numbered, admin-surface components (approvals, explainer, simulator) off the live path, real Postgres/Redis contents, revision snapshots, and the verifier sidecar; §4.5 split into two diagrams — the Compose MVP as it actually runs (five services, stdio subprocesses *inside* the gateway container, `:ro` policies + rw revisions submount, verifier sees only the public key) and the Phase-4 ALB/replicas target, with the stdio caveat updated from "SSE" to network/Streamable-HTTP wording; §4.6 data-flow redrawn with all 8 stages, the item-17 ABAC split (non-risk conditions at stage 4, `risk.*` after scoring before threshold mapping), every terminal feeding the signed audit write, and admin explain/simulate deliberately absent; §4.7 expanded into a real trust-domain discussion — why registration *is* the v1 trust decision, where that breaks (third-party / team-owned servers, drift history as signal), a `trust_tier`/`trust_score` YAML sketch marked SKETCH ONLY (loader rejects it today), trust as one more non-decayable risk factor plus a `server.*` ABAC attribute root, explicit v1 non-goals, item-29 cross-ref. All five mermaid blocks validated with `mermaid-cli`; inline mermaid only, no `docs/img` exports (sufficient per this item's scope; README finalization is item 28).
23. ~~Expanded adversarial test suite covering risk scoring, ABAC conditions (including missing-attribute cases, specifically inside `not(...)`), decision explanation accuracy, simulation accuracy, the approval-mismatch (TOCTOU) path, and `test_concurrent_audit_writes_do_not_collide`.~~ — **done** (tests only, no `services/` diff): items 16–21 were built test-first, so this item filled the *specific* §11 gaps rather than restarting coverage. New in `tests/adversarial/`: `test_risk_scoring.py` — the §11 canary end to end (high tier 30 + protected 30 + off-hours 25 = 85 → HUMAN_APPROVAL_REQUIRED, never ALLOW) with the business-hours factor kept IN for the first time (off-hours made deterministic by shrinking the configured window to empty, no clock mocking), plus the >90 band as a live DENY_RISK at clamped 100; `test_abac_adversarial.py` — missing attribute plain and inside `not(...)` through the real gateway → DENY_ABAC + a POLICY_ERROR audit row, mutation-checked (a temporary naive False-at-the-leaf in `abac.evaluate` flips exactly the `not(...)` case red, proving the test discriminates on the §4.8 inversion bug); `test_explanation_accuracy.py` — live-vs-GET reconstruction for the families the integration tests didn't cover (ALLOW, DENY_ABAC, DENY_VALIDATION, HUMAN_APPROVAL_REQUIRED, DENY_DRIFT) plus dry-run explain parity for the counter-independent ones via an identity with no live traffic; `test_simulation_accuracy.py` — the item-21 historical-hour claim proven by SQL-shifting two ALLOW rows to 03:00/12:00 UTC and hand-computing the 1/1 split a `context.hour >= 9` candidate must report (wall-clock replay could only produce 0/2 or 2/0); fixing it is future multi-replica work, out of scope here. TOCTOU needed nothing new — `test_risk_approval.py` already drives approve → mutated-args retry → DENY_APPROVAL_MISMATCH → original-args forward-once → reuse DENY_REPLAY. Hypothesis fuzzing (the other unimplemented §11 bullet) deliberately deferred — new dev dependency for a validator already edge-case unit-tested. **Phase 3 complete.**
